# 26.33 脚本加载阻断整改

> 测试环境：Windows 10 版 Minecraft Bedrock `1.26.33`
>
> 当前实机现象：没有 `[自定义NPC] 脚本已加载` 提示；右键 NPC 无 UI；皮肤始终呈现粗手臂；生成蛋没有 `NPC` 名称；实体只作为普通生物存在。
>
> 结论：**脚本模块未加载是第一阻断。未恢复脚本加载前，UI、皮肤属性同步、名称、AI、命令均不会工作。**

## 1. API 版本必须迁移到 1.26.33

当前项目仍在使用 1.26.10 的 API 依赖：

```json
"@minecraft/server": "2.6.0"
"@minecraft/server-ui": "1.3.0"
```

这与当前游戏 `1.26.33` 不匹配。经 npm 包注册表确认，1.26.33 对应版本为：

```json
"@minecraft/server": "2.9.0-beta.1.26.33-stable"
"@minecraft/server-ui": "2.2.0-beta.1.26.33-stable"
```

### 1.1 修改 BP manifest

文件：`BP/manifest.json`

将以下内容改为：

```json
"min_engine_version": [1, 26, 33]
```

```json
{
  "module_name": "@minecraft/server",
  "version": "2.9.0-beta.1.26.33-stable"
},
{
  "module_name": "@minecraft/server-ui",
  "version": "2.2.0-beta.1.26.33-stable"
}
```

不要保留 `2.6.0` 或 `1.3.0`。两个版本都必须同时替换。

### 1.2 修改本地开发依赖

文件：`package.json`

将开发依赖替换为：

```json
"devDependencies": {
  "@minecraft/server": "2.9.0-beta.1.26.33-stable",
  "@minecraft/server-ui": "2.2.0-beta.1.26.33-stable",
  "typescript": "^5.4.0"
}
```

然后在项目根目录执行：

```bash
npm install
npm exec tsc -- --noEmit
```

若 TypeScript 对 2.9.0 提示 API、事件或表单方法错误，必须按新声明修正源码，不得使用 `@ts-ignore` 压制错误。

## 2. 只做脚本加载最小验证

在恢复 UI、AI、皮肤或命令之前，先把 `BP/scripts/main.js` 暂时缩减到最小启动探针：

```js
import { world, system } from "@minecraft/server";

system.run(() => {
    world.sendMessage("[自定义NPC] 脚本已加载 | API 2.9.0 / UI 2.2.0");
});
```

此阶段不要 import `npc_interaction.js`、`npc_forms.js`、`npc_repository.js`、`skin_registry.js` 或其他业务模块。

### 2.1 验收条件

1. 重新打包并导入后，进入新测试世界。
2. 聊天栏必须出现且只出现一次：

```text
[自定义NPC] 脚本已加载 | API 2.9.0 / UI 2.2.0
```

3. Content Log 中不得有 script module、module version、import 或 JavaScript exception 错误。

没有通过这一步，不允许继续恢复其他代码。此时所有“右键无 UI”“皮肤不切换”都属于预期结果，因为脚本仍未运行。

## 3. 按依赖图逐个恢复脚本

最小启动探针通过后，再按此顺序恢复 import，每恢复一个模块就进游戏验证一次：

1. `skin_registry.js`
2. `npc_repository.js`
3. `npc_forms.js`
4. `npc_interaction.js`
5. 最后在 `main.js` 调用 `setupInteraction()`

每一步都保留启动探针。某一步恢复后探针消失或 Content Log 报错，该模块就是当前阻断点，先修它，不要继续恢复后续模块。

## 4. 右键 UI 验证

`@minecraft/server 2.9.0-beta.1.26.33-stable` 仍包含：

```js
world.beforeEvents.playerInteractWithEntity
```

事件字段仍是：

```js
const { player, target, itemStack } = event;
```

并且可以设置：

```js
event.cancel = true;
```

因此不要因为升级到 26.33 而改写成不存在的事件名或字段。

先在订阅回调中加入临时可见探针：

```js
world.beforeEvents.playerInteractWithEntity.subscribe((event) => {
    const { player, target } = event;
    if (target.typeId !== "customnpc:npc") return;

    event.cancel = true;
    player.sendMessage("[NPC] interact received");

    system.run(() => {
        player.sendMessage("[NPC] UI dispatch");
        // 这里再调用 openEditor 或 openDialogue
    });
});
```

验收顺序：

1. 右键先显示 `interact received`。
2. 下一 tick 显示 `UI dispatch`。
3. 再显示对应表单。
4. 如果第 1 条没有，检查实体是否真的为 `customnpc:npc`、脚本是否已加载。
5. 如果第 1 条有但第 2 条没有，检查 `system.run` 的异常。
6. 如果第 2 条有但没有表单，捕获 `form.show(player)` 的 rejection，聊天栏和 Content Log 同时输出完整 error。

## 5. 细手臂配置路径和正确流程

皮肤模型类型的唯一配置文件是：

```text
tools/skin_overrides.json
```

示例：

```json
{
  "1": 0,
  "2": 1,
  "15": 1
}
```

- `0`：Classic / Steve，4 像素粗手臂。
- `1`：Slim / Alex，3 像素细手臂。
- 键是皮肤槽位号：`npc_15.png` 对应 `"15"`。

不要手动长期修改 `BP/scripts/skin_registry.js`，因为它由检测工具生成，会被覆盖。

编辑 `skin_overrides.json` 后执行：

```bash
python tools/detect_skin_models.py
```

确认生成的 `BP/scripts/skin_registry.js` 中存在：

```js
15: { armModel: 1 }
```

`npc_2` 当前已经是 `armModel: 1`，并且 `RP/models/entity/npc_slim.geo.json` 的手臂尺寸已是 `[3, 12, 4]`。实机仍是粗手臂，说明客户端没有收到或没有应用 `customnpc:arm_model = 1`，最常见前置原因就是脚本未加载。

脚本恢复后，使用全新实体验证，不要复用旧 NPC：

```mcfunction
/summon customnpc:npc ~ ~ ~
```

选择 `npc_2` 或设定对应 skinId 后，必须验证：

- `customnpc:skin_id = 2`
- `customnpc:arm_model = 1`
- render controller 选择 `Geometry.slim`
- 正面、侧面和背面均为 3 像素宽手臂

## 6. 生成蛋没有名称

当前生成蛋源码已包含直接显示名：

```json
"minecraft:display_name": { "value": "NPC" }
```

资源包也应包含：

```text
RP/texts/languages.json
RP/texts/zh_CN.lang
RP/texts/en_US.lang
```

语言文件必须有：

```lang
item.customnpc:npc_spawn_egg.name=NPC
entity.customnpc:npc.name=NPC
```

如果物品栏仍无名称，不能继续修改语言键猜测。必须先确认游戏当前使用的就是最新 RP：

1. BP 与 RP 的 header/module/dependency 版本全部递增且一致。
2. 在游戏的全局资源包和行为包列表中删除旧“自定义NPC”包。
3. 重新导入新 `.mcaddon`。
4. 新建测试世界，同时启用新版 BP 和 RP。
5. 对游戏内已启用包的版本截图。
6. 在该新世界执行：

```mcfunction
/give @s customnpc:npc_spawn_egg 1
```

若仍无名称，提交 Content Log、物品 tooltip 截图、世界包设置截图和 `.mcaddon` 的归档文件清单。

## 7. 最终执行顺序

1. 升级 BP manifest 与 `package.json` 到 26.33 API 版本。
2. 执行 `npm install`、TypeScript 检查。
3. 用只含启动探针的 `main.js` 验证脚本加载。
4. 脚本探针通过后，按依赖图逐个恢复业务模块。
5. 用交互调试消息验证事件、延迟调度、表单三段链路。
6. 脚本运行后再验证 `arm_model` 和 Slim 模型。
7. 最后清理旧包、新建世界，验证生成蛋名称。

在聊天栏出现启动探针之前，不得声称 UI、皮肤、AI、名称或命令已修复。
