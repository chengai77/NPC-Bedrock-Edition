# 紧急整改：生成蛋名称、右键交互、Slim 手臂

> 触发原因：实机反馈确认以下三项均失败：生成蛋没有名称、右键 NPC 没有作用、细手臂皮肤仍以粗手臂渲染。
>
> 结论：当前包**没有通过最小可用验收**。停止继续添加功能，先按本文件修复并完成 V1-V8、V11 实机证据。

## 1. 已定位的直接问题

### P0-A：BP 与 RP 依赖版本不一致，资源包可能未被加载

**实际文件：**

- `BP/manifest.json:35`：BP 声明 RP 依赖版本为 `[1, 0, 5]`。
- `RP/manifest.json:7`：RP header 实际版本已为 `[1, 1, 0]`。

这不是可忽略的版本号问题。BP 依赖的 RP header UUID 虽然正确，但版本不匹配时游戏可以不加载当前资源包或命中旧缓存。结果会直接表现为：

- `.lang` 未进入客户端，因此生成蛋没有“NPC”名称。
- client entity / render controller 未生效，因此 Slim 属性变化仍显示旧 Classic 模型。
- 资源包资源与行为包脚本版本不是同一构建，任何渲染结论均不可信。

**程序员必须执行：**

1. 选择一个新版本，例如 `[1, 1, 1]`。
2. 将以下五处同步为完全相同的版本：BP header、BP data module、BP script module、BP 对 RP dependency、RP header、RP resource module。
3. 每次导入测试都递增版本，禁止复用同一版本覆盖导入。
4. 在游戏的全局资源包/行为包列表中删除旧的“自定义NPC”包，再重新导入新 `.mcaddon`；测试世界中也取消旧包并重新启用。
5. 在世界设置中确认 BP 与 RP 均显示为新版本，且 RP 确实已激活。
6. 导出 Content Log 中 manifest dependency 的所有相关行。

**自动验证器也必须修复：**

`tools/validate_pack.py` 当前只检查 module 与各自 header 的一致性，只检查 BP 是否含 RP UUID，**没有检查 BP 中该 UUID 的 dependency version 是否等于 RP header version**。补上该检查；不一致必须退出失败。

## 2. 生成蛋名称整改

当前源码中 `BP/items/npc_spawn_egg.json` 的 `minecraft:display_name` 已直接写为 `NPC`，BP/RP 的 `texts/languages.json` 和 `zh_CN.lang` 也存在。因此“没有名字”不能再按翻译文本缺失处理，优先按资源包未加载、旧包缓存或实际加载物品不是当前定义排查。

**必须按以下顺序实测：**

1. 完成上节版本同步与旧包清理。
2. 创建全新测试世界，明确同时启用新 BP 和新 RP。
3. 打开 Content Log，确认没有 item definition、items atlas、RP dependency 错误。
4. 执行：

```mcfunction
/give @s customnpc:npc_spawn_egg 1
```

5. 对物品栏中的物品截图，要求显示名严格为 `NPC`。
6. 若仍无名称，程序员必须提供：
   - Content Log 完整原文；
   - 游戏内物品 tooltip 截图；
   - 已导入包的版本截图；
   - 当前 `.mcaddon` 中 `BP/items/npc_spawn_egg.json`、`RP/texts/languages.json`、`RP/texts/zh_CN.lang` 的归档路径与原始内容。

不要再通过复制 `.lang` 文件或新增重复 atlas 文件来盲修。当前 canonical atlas 应仅保留 `RP/textures/items_texture.json`。

## 3. 右键交互整改

`BP/scripts/npc_interaction.js:19-36` 的交互处理依赖 `world.beforeEvents.playerInteractWithEntity`。右键毫无效果通常只可能是以下链路之一断了：脚本没有运行、事件订阅/字段错误、实体定义未加载、UI promise 被运行时异常中断。

### 3.1 先验证脚本是否真正运行

进入测试世界后，必须先确认聊天栏出现：

```text
[自定义NPC] 脚本已加载 | 版本: ...
```

没有这条启动探针时，禁止讨论交互组件或 UI。直接检查 Content Log 的 script entry、import、`@minecraft/server` / `@minecraft/server-ui` 版本错误。

### 3.2 用精确类型声明验证事件字段

本地安装的 `@minecraft/server 2.6.0` 声明确实包含 `beforeEvents.playerInteractWithEntity`。但不能仅依赖 `node --check`；程序员必须执行 TypeScript `checkJs`，确认回调事件对象字段是当前 API 声明中的字段。

检查 `node_modules/@minecraft/server/index.d.ts` 内 `PlayerInteractWithEntityBeforeEvent` 的接口定义后，修正 `npc_interaction.js` 的解构字段。当前代码使用：

```js
const { player, target } = event;
```

若声明中的目标实体字段不是 `target`，这一行会使 `target` 为 `undefined`，随后 `target.typeId` 直接抛错，UI 永远不会打开。不要猜字段名，按安装的 `2.6.0` 声明逐字实现。

### 3.3 增加临时调试探针，先证明事件再打开表单

在交互回调的第一行发送可见调试消息，并打印字段存在性。要求：

- 命中 `customnpc:npc` 时聊天栏先显示 `NPC interact received`。
- 延迟 `system.run` 执行前显示 `NPC UI dispatch`。
- `initializeNpc`、`openEditor`、`openDialogue` 任一失败时，聊天栏与 Content Log 同时输出完整异常。
- Form 的 promise 必须 `.catch()`，当前实现已有 catch，但必须确认异常可见。

调试通过后再删除或降级这些开发日志。

### 3.4 实体交互组件保持最小化

`BP/entities/npc.json:81-89` 的 `minecraft:interact` 应保留。不要在组件中配置原版事件来替代 Script API UI，也不要使用 `use_item: true` 消耗玩家手持物品。

**验收：**

- 创造模式右键：先收到事件探针，再打开编辑表单。
- 生存和冒险模式右键：先收到事件探针，再打开对话表单。
- 对未配置对话的 NPC，显示“还没有设置对话”，而不是无反馈。
- Content Log 无未处理异常。

## 4. Slim 仍为粗手臂的整改

源码已经存在以下选择链：

- `BP/scripts/skin_registry.js:8`：`skinId=2` 映射 `armModel=1`。
- `BP/scripts/npc_repository.js:195-196`：初始化写入 `customnpc:arm_model` client property。
- `RP/entity/npc.entity.json:126`：读取该 property 到 `v.arm_model`。
- `RP/render_controllers/npc.render_controllers.json:36`：`v.arm_model ? Geometry.slim : Geometry.classic`。

既然实机仍显示粗臂，优先判断“新 RP 未加载”而不是继续修改 registry。P0-A 版本不一致是第一阻断。

资源包确认已加载后，按下面顺序定位：

1. 用全新 `/summon customnpc:npc ~ ~ ~` 生成实体，避免旧实体的数据误导。
2. 选择 skin 2 后，记录 `customnpc:skin_id` 和 `customnpc:arm_model` 的实际客户端同步值；必须为 `2` 和 `1`。
3. 在 render controller 中临时将 `Geometry.slim` 与 `Geometry.classic` 选择结果用明显可区分的调试 geometry 或日志替换，确认客户端实际进入哪一分支；验证后恢复标准 geometry。
4. 检查 `RP/models/entity/npc_slim.geo.json` 的骨骼尺寸、origin 和 UV：细手臂必须为 3 像素宽，且左右手臂的 origin 必须同时为 Slim 设计；只将 cube size 从 4 改成 3 不足以正确对齐。
5. 用 skin 1（Classic）和 skin 2（Slim）分别正面、背面、两侧截图。必须同时验证基础层和袖子第二层。

**禁止事项：**

- 不要让玩家在表单中直接选择 armModel。
- 不要让动态属性中的旧 `arm_model` 覆盖 skin registry 的结果。
- 不要凭 PNG 自动检测结果默默回退 Classic；有歧义时要求 `skin_overrides.json` 明确指定。

## 5. 必须修改的验证顺序

按以下顺序提交，而不是一次性声称全修复：

1. 同步 manifest 版本并扩展 `validate_pack.py` 的 BP→RP version 检查。
2. 输出自动验证结果，必须失败/通过可复现。
3. 清理旧导入包，创建全新测试世界，完成 V1-V6。
4. 修正并验证交互事件字段，完成创造、生存、冒险三种模式的 V7-V8。
5. 确认新 RP 正在被加载后，再验证 Classic/Slim 的 V11。
6. 提交 Content Log、测试截图、构建版本和最终 `.mcaddon`。

在 V3、V7、V11 同时有真实证据前，不接受“生成蛋、右键、细手臂已修复”的结论。
