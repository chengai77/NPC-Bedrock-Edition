# 自定义 NPC 实施交接

> 目标版本：Minecraft Bedrock 1.26.10
>
> Script API：`@minecraft/server` 2.6.0、`@minecraft/server-ui` 1.3.0
>
> 项目：`D:\封装\草方块社区\自定义npc`
>
> 当前角色分工：你负责实现、测试与提交证据；技术总监负责后续审查。
>
> 发布结论：**在完成本文全部 P0 项和实机验收前，不得发布或宣称可用。**

## 1. 目标与边界

请保留并完成以下最终能力：

- 创造物品栏和命令均可获取显示名严格为“NPC”的生成蛋。
- `/give @s customnpc:npc_spawn_egg 1` 可用，生成蛋可生成 `customnpc:npc`。
- 创造模式玩家右键 NPC 打开编辑界面；生存、冒险模式玩家右键只打开对话界面。
- 每个 NPC 独立保存名称、皮肤、手臂模型、对话、受控命令和 AI 开关；重进世界后仍存在。
- 保留 `npc_1.png` 至 `npc_100.png` 共 100 个皮肤槽位，并按数字顺序显示。
- `npc_1` 固定名称“作者”，`npc_2` 固定名称“星野”；其余皮肤允许编辑名称。
- 皮肤必须使用标准 64x64 玩家模型，完整支持 Classic 粗臂、Slim 细臂和全部第二层。
- AI 关闭时 NPC 原地静止；开启时自然行走，名称不追加 `[AI]`、`[静止]` 等状态字样。

不要恢复 NPC 方块、NPC 法杖或基于原版 `minecraft:npc` 的旧方案。项目必须以 `customnpc:npc` 自定义实体作为唯一核心。

## 2. 当前已知事实

以下仅为静态检查或源码阅读结论，不等同于游戏内已通过：

- 行为包 manifest 已声明 `@minecraft/server 2.6.0`、`@minecraft/server-ui 1.3.0`，并由 BP 单向依赖 RP。
- 生成蛋定义为 `customnpc:npc_spawn_egg`，使用 `minecraft:entity_placer` 生成 `customnpc:npc`。
- 实体定义已有 `minecraft:interact`，语言文件已有 `texts/languages.json` 注册入口。
- 资源包已有 Classic / Slim 两套 geometry、100 个纹理声明和 `entity_alphatest` 材质。
- 当前脚本 AI 仍使用周期 `teleport()`；这不是自然行走，且当前 walk 动画条件依赖 `q.modified_move_speed`，大概率无法在 teleport 下正常触发。
- 当前物品 `format_version` 为 `1.20.20`，实体为 `1.26.10`。是否能在目标游戏正常注册，必须以 Content Log 和实机命令验证为准。

## 3. P0：必须先完成

### P0-1 修复注册链路并先做最小实机验证

实施顺序必须如下，未通过前不要调 UI、皮肤或 AI：

1. 用开发包目录部署 BP 与 RP，确认世界同时启用两个包。
2. 删除旧导入包和测试世界的旧缓存，导入递增版本的新包。
3. 打开 Content Log，确认无 manifest、物品、实体、脚本入口、客户端实体和 render controller 报错。
4. 执行：`/give @s customnpc:npc_spawn_egg 1`。
5. 执行：`/summon customnpc:npc ~ ~ ~`。
6. 用生成蛋生成 NPC。
7. 保存完整 Content Log 原文和三张对应截图：命令成功、物品图标与名称、实体生成。

注意：命令中不得混入引号、尖括号、中文全角冒号或聊天框自动补全残留。正确命令仅为：

```mcfunction
/give @s customnpc:npc_spawn_egg 1
```

如果 `/give` 仍失败，优先检查 Content Log 中该物品 JSON 的第一条错误，而不是继续修改命令文本。

### P0-2 重做 AI 为实体原生行走

当前 `BP/scripts/main.js` 使用 `npc.teleport()` 每 20 tick 平移，必须移除作为常规 AI 的实现。

采用实体组件组实现唯一的 AI 方案：

- 常驻组件只负责实体基础属性、物理和交互。
- `customnpc:ai_enabled` 组件组包含 `minecraft:movement`、`minecraft:navigation.walk`、`minecraft:behavior.random_stroll`，以及必要的避障/跳跃组件。
- `customnpc:ai_disabled` 状态不得有移动目标或随机游走行为。
- 编辑菜单仅通过实体事件添加或移除 AI 组件组；不要以名字、tag 文本或脚本内存 Map 作为 AI 事实来源。
- NPC 的 AI 开关必须持久化；世界重载后按保存状态恢复组件组。

动画要求：

- 行走动画由真实移动速度驱动。
- 左右腿反向摆动，左右臂与腿反向摆动。
- 静止时不触发行走动画。
- 禁止用 teleport 模拟行走，也不要只摆动身体。

验收场景：平地、坡地、墙边、水边、连续启停、两个玩家同时观察。不得出现滑行、跳格、穿墙、持续抖动或名称变化。

### P0-3 将手臂模型改为皮肤槽位的派生状态

`arm_model` 不能由表单、旧动态属性或任意保存数据决定。它必须始终由 `skin_id` 对应的皮肤 registry 推导。

实现要求：

1. 建立一个唯一皮肤 registry，至少包含：槽位号、皮肤文件名、模型类型、固定名称规则。
2. `skin_id -> arm_model` 的映射在保存、加载、初始化、迁移四处统一调用同一个函数。
3. `arm_model` 可作为客户端同步缓存，但不能作为真实来源。
4. 扫描已加载的旧 NPC：发现皮肤与手臂模型不匹配时，仅修正模型属性，不得覆盖名称、对话、命令或 AI 状态。
5. PNG 本身没有 Classic/Slim 元数据。自动检测结果歧义时，构建必须失败，直到 `skin_overrides.json` 提供显式模型类型；禁止默认回退到 Classic。
6. `npc_1`、`npc_2` 必须在 registry 中显式设定模型类型，不依赖自动猜测。

验收：故意制造 `skin_id=2` 搭配粗臂缓存的旧数据，加载后必须自动恢复为 Slim；`skin_id=1` 仍必须为 Classic。

### P0-4 命令功能必须改为严格白名单

现有“禁止若干前缀”的黑名单不足以阻断间接命令。以下例子必须被拒绝保存：

```mcfunction
execute as @s run op <name>
function namespace:dangerous
schedule function namespace:dangerous 1t
```

实施要求：

- 删除黑名单方案。
- 建立严格允许的命令类型、参数数量、参数格式、最大长度和执行身份。
- 第一版仅允许明确需要的少量命令；未定义的命令一律拒绝。
- 必须拒绝 `execute`、`function`、`schedule`、`scriptevent`、`op`、`deop`、`permission`、`reload`、`gamerule`、`kick`、`ban` 及其他未列入白名单的命令。
- 玩家数据只能通过唯一占位符 `{player}` 注入；不得将整条命令内的 `@p` 替换为玩家名。
- 保存时完成解析与校验；普通玩家只能触发已保存、已校验的命令，不能提交命令文本。
- 新建 `docs/COMMAND_POLICY.md`，写清允许命令、允许参数、执行身份、拒绝规则和测试样例。

## 4. P1：功能稳定性

### P1-1 既有实体迁移与持久化

- 新生成实体、世界重载后已加载实体、脚本升级前的旧实体，必须共用同一初始化/迁移入口。
- 启动扫描必须分批限量执行，不能在单 tick 扫描整个世界。
- 数据持久化不得仅依赖 JavaScript `Map`；名称、皮肤、对话、命令和 AI 状态必须保存在实体可持久化数据中。
- 所有读取数据必须先校验并夹紧范围，再同步到客户端属性。

### P1-2 表单与数据上限

必须为以下数据配置明确上限：名称、对话正文、对话按钮、命令、说明、对话条数、命令条数、最终 JSON 字节长度。

- 保存超限时显示具体失败原因。
- 皮肤选择改为分页，每页最多 20 项，且按数字 `1..100` 排序。
- 取消表单或无效实体时不修改数据、不抛未捕获异常。
- 创造编辑和生存对话必须是两条独立权限路径。

### P1-3 渲染与皮肤资产

- Classic 和 Slim 均须包含头、身体、双臂、双腿的基础层，以及帽子、夹克、左右袖、左右裤腿等第二层。
- 使用 `entity_alphatest`，第二层透明像素不得变为色块。
- client entity、render controller、geometry、animation、纹理 100 项之间的引用必须全部可解析。
- 对 `npc_1` 和 `npc_2` 的实机渲染做正面、侧面、背面、移动、静止截图。

## 5. 必须补齐的自动验证

完善 `tools/validate_pack.py` 或新增验证脚本，至少失败检查以下情况：

- BP/RP manifest、生成蛋、实体、脚本入口、client entity、items atlas、生成蛋 PNG 缺失。
- 脚本入口或相对 import 缺失。
- client entity 引用的 geometry、render controller、animation、texture key 不存在。
- render controller 声明的 100 个皮肤未全部在 client entity 和磁盘纹理中对应。
- 皮肤数量不是精确 100，文件名不严格为 `npc_1.png` 至 `npc_100.png`。
- 皮肤大小不是 64x64。
- 皮肤模型检测为歧义但未提供显式 override。
- `.mcaddon` 内含 `node_modules`、历史目录或不应发布的开发文件。

自动验证通过只能说明引用图和归档结构正确，不能替代游戏实测。
