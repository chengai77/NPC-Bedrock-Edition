## 2. P0：发布阻断问题

### P0-1 命令安全策略可被间接命令绕过

**位置：** `BP/scripts/npc_forms.js:9-19`、`BP/scripts/npc_forms.js:241-250`

当前只拦截 `op `、`stop`、`gamerule ` 等开头文本。它不能防止：

```mcfunction
execute as @s run op <name>
function <namespace>:dangerous
schedule function <namespace>:dangerous 1t
```

这些命令不以被拦截字符串开头，但仍能造成权限提升、调用高权限函数或延迟执行危险逻辑。当前执行方式是 `npc.dimension.runCommandAsync(cmd)`，影响面不是单个玩家。

**整改指令：**

1. 删除黑名单模式 `BLOCKED_PREFIXES`，改为严格允许列表。
2. 第一期只允许明确业务所需的少量命令，例如 `say`、`tellraw`、`give`、`effect`、`tag`、`scoreboard players`；其他一律拒绝。
3. 禁止 `execute`、`function`、`schedule`、`scriptevent`、`op`、`deop`、`permission`、`reload`、`gamerule`、`kick`、`ban` 等命令及其任何间接形式。
4. 不得把整条命令里的 `@p` 文本替换成玩家名称。改用唯一占位符，例如 `{player}`，只替换该占位符，并对替换值使用 Bedrock 选择器/名称所需的转义规则。
5. 对每一条保存时的命令执行解析：验证第一个 token 位于允许列表、参数数量受限、长度受限。
6. 新增 `docs/COMMAND_POLICY.md`，列出允许命令、参数格式、执行身份与拒绝理由。

**验收：** 创造编辑者输入上面 3 条绕过示例以及任何未知命令，均无法保存；普通对话只能执行已验证的允许列表命令。

### P0-2 皮肤自动粗细臂映射不是强制不变量

**位置：** `BP/scripts/npc_repository.js:64-65`、`95`、`109-119`、`140`、`147`

当前 `validateNpcData()` 接受传入的 `data.armModel`，`saveNpc()` 也直接保存它。结果是：即使 `skinId=2` 的 registry 指定 Slim，旧实体数据、损坏数据或未来调用者仍可写入 `armModel=0`，重新出现“细手臂皮肤套到粗手臂模型”。

**整改指令：**

1. `armModel` 不再视为独立可编辑或可信持久化数据。
2. 在 `validateNpcData()` 中始终执行：`safe.armModel = getArmModel(safe.skinId)`。
3. 在 `loadNpc()` 中也根据最终 `skinId` 计算 `armModel`，不读取旧值作为有效输入。
4. 在 `saveNpc()` 中只保存由 registry 派生的值；旧 dynamic property 可保留为缓存，但不可作为事实来源。
5. `initializeNpc()` 必须无条件将客户端同步属性设置为 registry 的结果，处理旧实体和版本升级。
6. 增加一次性迁移：扫描加载区 NPC，发现 `skin_id` 和 `arm_model` 不匹配时只修正 `arm_model`，不得改动用户名称/对话/命令。

**验收：** 手动将任意现有 NPC 的 `arm_model` 设错或用旧世界数据导入，首次加载后皮肤 2 仍渲染为 Slim，皮肤 1 仍渲染为 Classic。

### P0-3 AI 现在是周期 teleport，客户端不会产生真正行走动画

**位置：** `BP/scripts/main.js:80-92`、`RP/entity/npc.entity.json:133`、`RP/animations/npc.animation.json:13-20`

AI 每 20 tick teleport 一次，实体定义的 `minecraft:movement` 又为 `0.0`。`q.modified_move_speed > 0.01` 很可能始终为 false，因此 walk 动画不会播放；即使播放，动画目前也只旋转 body，双腿和双臂没有摆动。视觉结果会是滑行/跳格，不符合自然行走要求。

**整改指令：**

1. 决定唯一移动方案，不要同时混用普通 AI 与 script teleport。
2. 推荐方案：实体组件组控制 `minecraft:movement`、`minecraft:navigation.walk`、`minecraft:behavior.random_stroll`；AI 开关仅通过实体事件添加或移除该组件组。
3. 若必须保留脚本移动，新增 client-sync `customnpc:is_walking` 属性，在移动状态显式写入，渲染动画据此播放；同时在到达、碰撞、暂停、关闭 AI 时复位。
4. `npc.animation.json` 的 walk 必须为双腿反向摆动、双臂反向摆动；禁止只让 body 静止。
5. movement 不得固定为 `0.0` 后声称使用原版行走。
6. 增加坡地、障碍物、水边、多人视角下的移动回归测试。

**验收：** 关闭 AI 时完全静止；开启后 NPC 在合法地面上自然迈步，不穿墙、不原地抖动、不滑行、不改写名字。

## 3. P1：高优先级功能与稳定性问题

### P1-1 旧 NPC 只在生成或交互时初始化

**位置：** `BP/scripts/main.js:103-108`、`BP/scripts/npc_repository.js:125-149`

仅监听新生成事件。世界升级或脚本升级后的既有 NPC，在玩家右键前不会同步名称、皮肤和模型属性；AI 扫描又只调用 `loadNpc()`，不会修正客户端属性。

**整改指令：** 在启动后延迟扫描各维度已加载的 `customnpc:npc`，以限额分批方式调用迁移/初始化；不得在一 tick 全世界全量扫描。后续 `entityLoad` 或实体生成路径也应共用同一迁移函数。

### P1-2 对话与命令缺少数量和长度上限

**位置：** `BP/scripts/npc_repository.js:79-87`、`BP/scripts/npc_forms.js:99-118`、`150-171`

动态属性是有限长度字符串，但当前无限添加对话、命令和文本长度，最终会在 `JSON.stringify()` 或 `setDynamicProperty()` 抛错。玩家也可创建巨型 100 项 Action Form。

**整改指令：**

- 限制 NPC 名称、对话文本、按钮文本、命令、说明的字符长度。
- 限制对话条数、命令条数，并在达上限时禁用添加入口。
- 保存前检查最终 JSON 字节长度；拒绝保存时显示具体原因。
- 皮肤选择 100 项改为分页，例如每页 20 项，避免低端客户端表单过长。

### P1-3 `detect_skin_models.py` 对自定义皮肤存在可预期误判

**位置：** `tools/detect_skin_models.py:43-50`

用两个“未使用 UV 区域是否透明”判断 Classic/Slim，纯透明袖子、极简皮肤或特殊外层会得到 `AMBIGUOUS`；当前仍自动回退 Classic。该行为会悄悄让用户替换的 Slim 皮肤出错。

**整改指令：**

1. 发现 `AMBIGUOUS` 时构建应失败，除非 `skin_overrides.json` 已显式配置该槽位。
2. 将 `skin_overrides.json` 定义为正式资产清单，不要当作可选临时文件。
3. 在 `SKIN_SLOT_GUIDE.md` 写明：覆盖 PNG 后必须运行检测工具；若输出 ambiguous，必须填写 `{ "槽位号": 0 或 1 }`。
4. 对 `npc_1` 和 `npc_2` 保留显式覆盖，检测仅作一致性提示。

### P1-4 物品与实体还没有 schema/实机加载证明

**位置：** `BP/items/npc_spawn_egg.json:1-19`、`BP/entities/npc.json:1-56`、`docs/TEST_MATRIX.md:7-20`

项目历史已出现过 `/give` 无法识别物品。当前 `format_version: "1.20.20"`、`minecraft:display_name`、`menu_category` 和 `entity_placer` 只是静态存在，尚未被 1.26.10 Content Log 证明。

**整改指令：** 先只测试 V1-V6，保存完整 Content Log。没有 V3 `/give`、V5 `/summon`、V6 生成蛋生成实体的真实证据，不得进入 UI、皮肤或 AI 验收。

### P1-5 验证工具不验证所有运行时引用

**位置：** `tools/validate_pack.py:18-103`

工具能检查 JSON、皮肤和图集，但没有验证：客户端实体的 geometry、render controller、animation、texture key 是否逐一存在；也没有验证脚本入口和 import 图是否完整。

**整改指令：** 扩展工具以验证：

- `RP/entity/npc.entity.json` 里的每个 geometry、animation、render controller 均在对应文件中定义。
- render controller 的 100 个 `Texture.skin_N` 全部在 client entity 声明。
- 每个 texture 路径均存在 PNG。
- `BP/manifest.json` 的 script entry 存在，所有相对 import 存在。
- 压缩包必须仅含 BP/RP，且包含 BP/RP manifest、主脚本、实体、物品、client entity、items atlas、生成蛋 PNG。

## 4. P2：质量与维护性优化

### P2-1 删除无使用动画或接入编辑状态

**位置：** `RP/animations/npc.animation.json:22-33`

`animation.npc.editing` 未在客户端实体中引用。删除它，或新增明确的 client-sync 编辑状态后接入；不要保留死资源。

### P2-2 修正检测工具的 Pillow 过时 API 警告

**位置：** `tools/detect_skin_models.py:30`、`40`

实际运行出现 Pillow `getdata()` deprecation warning。改用 Pillow 新 API，确保工具升级后不失效。

### P2-3 清理代码中的无效变量与过宽 ignore

**位置：** `BP/scripts/npc_repository.js:129`、`BP/scripts/main.js:113`

`hasName` 未使用。`@ts-ignore` 应替换为可证明安全的 Player 类型守卫或安全 helper，避免屏蔽未来 API 错误。

### P2-4 版本号与发布流程

当前所有 manifest 为 `1.0.5`，但最终发布前必须在 BP header、BP modules、BP 对 RP dependency、RP header、RP module 同步提升。每次导入测试均记录测试包版本，避免把缓存问题误判为逻辑问题。

## 5. 给程序员的执行顺序

1. 完成 P0-1 命令白名单并编写 `COMMAND_POLICY.md`。
2. 完成 P0-2：把 `armModel` 改为由 `skinId` registry 唯一派生，并加入旧实体迁移。
3. 完成 P0-3：选择单一 AI 移动方案，实现真实行走动画与 AI 开关。
4. 完成 P1-1/P1-2：既有实体迁移、数据上限、皮肤分页。
5. 完成 P1-3/P1-5：让检测和验证工具在歧义或引用缺失时失败。
6. 先执行并记录 V1-V6 实机结果与 Content Log。
7. V1-V6 全部通过后，再执行 V7-V14。
8. 提交整改后只提交源文件、验证工具、Content Log、V1-V14 实测证据；不得只提交新的 `.mcaddon`。

## 6. 当前审查结论

- **可保留：** BP→RP 单向依赖、代码模块拆分、生成蛋/实体双入口、客户端双 geometry、`entity_alphatest`、构建期 skin registry、静态验证和可复现打包。
- **不可批准：** 游戏内尚无一项 V1-V14 已验证；P0 命令策略、自动粗细臂强制映射、AI 自然移动尚未满足发布要求。
